Documentation

This extension uses the Web-Request-API.
https://developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/API/webRequest https://developer.chrome.com/extensions/webRequest

A description of the types of web requests can be found here:
https://developer.mozilla.org/en-US/Add-ons/WebExtensions/API/webRequest/ResourceType https://developer.chrome.com/extensions/webRequest

Processing (changing / blocking) of web requests and headers is done in the following way:

Event "browser.webRequest.onBeforeRequest"

  • HTTP redirect

    Settings in section redirecting HTTP requests
    In case of a match the requested URL is changed and a HTTP redirect is performed.

  • blocking / allowing

    If the URL is secure the web request is allowed. Otherwise:
    Settings in section blocking web requests in the following order

    1. Option "allowed based on type"
    2. Option "blocked based on type"
    3. Option "FONT blocked based on name"
    4. Option "allowed for all types"
    5. Option "blocked for all types"
    6. If none of the options above matched : the request is allowed

Event "browser.webRequest.OnBeforeSendHeaders"

  • Referer

    If the URL is secure the referer is allowed. Otherwise:
    Settings in section blocking request headers -- Referer
    1. allow Referer
    2. block Referer
    3. If none of the options above matched : Referer is allowed

  • Cookies

    If the URL is secure Cookies are allowed. Otherwise:
    Settings in section blocking request headers -- Cookies
    1. allow Cookies
    2. block Cookies
    3. If none of the options above matched : Cookies are allowed

  • User Agent

    Settings in section changing request headers -- User Agent
    The user agent is changed based on the rules found in the section "User Agent". If no rule applies User Agent is not changed.

Event "browser.webRequest.onHeadersReceived"

  • Cookies

    If the URL is secure Cookies are allowed. Otherwise:
    Settings in section blocking response headers -- Cookies
    1. allow Cookies
    2. block Cookies
    3. If none of the options above matched : Cookies are allowed

  • Content-Security-Policy

    If the URL is secure no CSP header is inserted. Otherwise:
    Settings in section changing response headers -- Content-Security-Policy
    1. whitelist
    2. blacklist
    3. If none of the options above matched : no CSP header is inserted

changing response headers

Content-Security-Policy Header

whitelist

use newlines to separate the values, do not use spaces.
No CSP header will be added to the response headers for all requests of type MAIN_FRAME or SUB_FRAME matching one of the URL strings.


blacklist

Use newlines to separate pairs of values (URL string / CSP header string), the first space separates the values.
A CSP header will be added to the response headers for all requests of type MAIN_FRAME or SUB_FRAME matching one of the URL strings unless the URL matches one of the strings in the whitelist.
More information about blocking content using CSP can be found here:
https://developer.mozilla.org/en-US/docs/Web/HTTP/CSP
If you want the browser not to execute SCRIPT by default, use the following entry for blocking SCRIPT in all web pages:
/ script-src 'none';report-uri https://spwebfilter.invalid/;report-to https://spwebfilter.invalid/
and use the white list for exceptions. Please expect that the browser will not evaluate <noscript> sections of the web page as this setting does not disable Javascript in the browser settings.


redirecting HTTP requests

Replace Strings

Use newlines to separate pairs of values (old value / new value), use one space to separate values.
Redirecting of SCRIPT, FONT, STYLESHEET etc might not work.
The first match is executed, all other pairs are ignored.
The redirected URL will be checked against the rules. Do not create infinite loops.
